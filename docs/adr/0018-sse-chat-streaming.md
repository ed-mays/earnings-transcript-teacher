# SSE for Chat Streaming

**Status:** Accepted
**Date:** 2026-03-26

## Context

The Feynman learning chat streams AI responses token-by-token as they're generated by Perplexity (ADR 0013). Users expect to see text appearing progressively, not waiting for the complete response. The streaming transport needed to work with FastAPI's response model, the Perplexity API's streaming output, and the Next.js frontend's fetch API.

## Decision

Use Server-Sent Events (SSE) for streaming chat responses from the FastAPI backend to the frontend. The implementation uses FastAPI's `StreamingResponse` with `text/event-stream` content type. The backend's `stream_chat()` function yields chunks from Perplexity's streaming API, which are forwarded as SSE events to the client.

The key driver was simplicity — SSE is unidirectional (server → client), which exactly matches the chat pattern (user sends a message via POST, server streams the response back). SSE works natively with HTTP/2, requires no connection upgrade negotiation (unlike WebSockets), and is trivially compatible with both FastAPI's streaming model and the browser's `EventSource` / `fetch` with `ReadableStream` APIs.

## Alternatives considered

1. **WebSockets** — Bidirectional communication with persistent connections. Rejected because: (a) the chat pattern is request-response (user sends message, server streams reply) — bidirectional communication is unnecessary, (b) WebSockets require connection upgrade negotiation and persistent connection management on the server, (c) WebSocket connections don't go through HTTP middleware (auth, rate limiting) without additional configuration, and (d) Railway's container hosting handles HTTP connections more naturally than persistent WebSocket connections.

2. **Long polling** — Client repeatedly polls an endpoint until the response is complete. Rejected because: (a) long polling is inefficient for token-by-token streaming (each poll returns a small payload with high overhead), (b) it requires server-side state to track partial responses, and (c) the user experience is noticeably worse than true streaming (chunks arrive in batches rather than continuously).

3. **gRPC streaming** — Efficient binary streaming protocol. Rejected because: (a) gRPC requires a separate server and client setup (not native to FastAPI or browser JavaScript), (b) browser gRPC support requires grpc-web proxy, adding infrastructure complexity, and (c) the text-based nature of chat responses doesn't benefit from gRPC's binary encoding efficiency.

4. **Chunked transfer encoding (raw HTTP streaming)** — Streaming via HTTP response body without SSE framing. Rejected because: (a) SSE provides structured event framing (`data:`, `event:`, `id:` fields) that makes parsing robust, (b) SSE has built-in reconnection support, and (c) the overhead of SSE framing is negligible for text chat responses.

## Consequences

**Easier:**
- Trivial server implementation — FastAPI's `StreamingResponse` with an async generator
- Standard browser APIs — `fetch()` with `ReadableStream` or `EventSource`
- Works through HTTP proxies and load balancers without special configuration
- No connection state management — each chat exchange is an independent HTTP request/response

**Harder:**
- SSE is unidirectional — if bidirectional streaming is needed in the future (e.g., user interrupting a response mid-stream), WebSockets would be required
- SSE connections can be dropped by intermediary proxies that don't support long-lived HTTP connections (mitigated by Railway's direct container access)
- No built-in binary data support — SSE is text-only (not a limitation for chat text)
- Browser `EventSource` API has limited error handling compared to WebSocket's `onerror`/`onclose` callbacks
